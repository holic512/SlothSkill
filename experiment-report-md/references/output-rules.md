# Output Rules

All reports generated by this skill must follow these rules unless the user gives a stricter requirement that still fits the evidence.

## Core Rules

- output the final Markdown draft, not prompt notes
- write in English
- place the source draft at `docx/<report-name>/report.source.md`
- keep requirements grounded in the topic and experiment goal
- keep design and implementation grounded in repository evidence
- avoid AI-sounding, prompt-sounding, or analysis-sounding narration

## Heading Rules

Use Markdown heading depth to match the real chapter depth.

- `#` for chapters
- `##` for sections
- `###` for subsections

Do not fake headings with bold text.

## Diagram Rules

- include `plantuml` blocks only when a diagram is justified
- prefer sequence diagrams or interaction diagrams when they clarify runtime behavior
- route professional diagram production to `plantuml-professional-diagrams`
- do not claim that diagrams were rendered unless that skill has actually done it

## Evidence Rules

- every major section must be supportable by the repo, the task statement, or both
- do not invent screenshots, APIs, tables, or test outcomes
- when screenshots are missing, label them as placeholders
- when test execution is missing, write a validation plan or test design

## Style Rules

Avoid phrases such as:

- "according to code analysis"
- "based on repository analysis"
- "the following report is generated"
- any wording that exposes prompt mechanics

Prefer direct technical prose that reads like a real report.
