# 输出规范

所有报告统一遵循以下规则；若用户显式指定格式要求，则在不破坏真实性的前提下兼容用户要求。

## 1. 核心原则

- 输出最终报告，不输出中间提示词
- 以“题目与任务目标”驱动需求分析，以“目录与代码证据”支撑设计、实现和测试
- 产物统一落在 `docx/<报告名>/`
- 生成完整交付版报告前，必须已具备 `java`、Graphviz `dot`、`pandoc`
- Word 导出时优先从 `assets/docx-templates/manifest.json` 选择参考模板；未给样式条件时使用通用模板
- 不得出现 AI 口吻、提示词痕迹、模板话、待补充字样

## 2. 双阶段文档规则

交付过程分为两个 Markdown 阶段：

- `report.source.md`
  - 保留 PlantUML 代码块
  - 用于图片渲染和排错
- `report.md`
  - 将 PlantUML 代码块替换为图片引用
  - 用于最终阅读和导出 DOCX

如果缺少 `java`、Graphviz `dot` 或 `pandoc`，本 skill 应立即停止，并提示用户先安装依赖，而不是降级输出。

## 3. 标题与结构

- 严格沿用所选模板的章节结构
- 不新增与项目证据无关的大章节
- 最多使用模板允许的标题层级
- 需求分析章节不得写成代码功能清单
- Markdown 标题层级必须与文档章节层级严格对应

固定映射规则：

- `#`：章级标题
  - 例如 `# 第1章 需求分析`
  - 例如 `# 一、需求分析`
- `##`：节级标题
  - 例如 `## 1.1 背景与目标`
  - 例如 `## （一）系统开发背景与意义`
- `###`：小节标题
  - 例如 `### 1.1.1 用户角色`
  - 例如 `### 1. 功能模块划分`

明确禁止：

- 把“第1章”“第一章”“一、需求分析”写成 `##`
- 把二级节标题写成 `#`
- 用加粗正文代替真实 Markdown 标题

## 4. 图表与代码编号

- 图、表、代码编号必须连续
- 每张图表必须在正文中有引用
- 图标题放在图下方，表标题放在表上方
- 如果模板要求章节内编号，如 `图2-1`，则按模板执行
- 如果模板要求全文连续编号，如 `图1`、`图2`，则保持全文一致

## 5. PlantUML 规范

源码版 Markdown 中的 PlantUML 代码块默认带：

```plantuml
@startuml
skinparam monochrome true
' 具体图内容
@enduml
```

最终交付版 Markdown 中：

- 优先使用渲染后的图片
- 图片路径统一使用相对路径 `images/...`
- 默认不再保留原始 PlantUML 代码块

## 6. 截图占位规则

当缺少真实截图时：

- 使用“图X 某界面截图（占位）”作为图题
- 紧接着写该界面的功能说明和操作要点
- 不得把占位图描述写成已经真实截取

## 7. 文案表达限制

禁止出现这些表达：

- “根据当前仓库中的真实代码实现”
- “根据代码分析可知”
- “结合仓库内容可见”
- “下面根据要求生成”
- 任何明显的提示词腔、分析腔、AI 腔连接句

允许的自然写法示例：

- “系统面向课程设计中的业务场景进行建模”
- “系统采用分层结构组织核心业务逻辑”
- “测试部分围绕功能目标与异常场景进行设计”

## 8. 表格与说明写法

- 表格要服务于真实内容，不为凑数堆表
- 字段、模块、测试用例、工具清单等都要尽量对应题目与项目证据
- 如果字段明细不完整，写已识别到的部分，并保持表头规范

## 9. 依赖要求

- 没有 `java`：停止执行，并提示安装 Java
- 没有 Graphviz `dot`：停止执行，并提示安装 Graphviz
- 找不到 PlantUML JAR：停止执行，并提示检查 skill 资源
- 没有 `pandoc`：停止执行，并提示安装 pandoc
- 单张 PlantUML 渲染失败：停止执行，并返回该图的具体错误

## 10. 生成后的自检清单

- 标题层级合法
- 章级标题确实使用 `#`，不是 `##`
- 图表编号连续
- 图表在正文均被引用
- 章节没有缺失
- 需求分析以题目和目标为主，不是代码说明书
- 没有 AI 口吻、模板话、待补充、分析腔
- 内容与题目、目录证据一致
