# 目录扫描清单

生成报告前，至少检查以下信息位。目标是把“题目驱动的信息”和“代码证据驱动的信息”分开。

## 1. 题目来源

优先识别报告题目、课程名称、项目名称与任务目标：

- 用户请求中的题目与课程名
- 目录名称
- 任务书、实验要求、课程说明
- 旧提示词标题、旧报告标题
- README、封面页、文档标题

如果题目不完整，可以归纳，但不要生造不存在的具体业务细节。

## 2. 项目基础信息

- 子目录结构、命名风格
- README、说明文档、日志、开发记录
- 旧报告、提示词、模板文件

## 3. 技术栈与运行环境

- 编程语言与框架
- 构建工具、包管理工具、依赖声明
- 配置文件与环境变量文件
- 运行入口、启动脚本、部署脚本

## 4. 模块与功能证据

- 控制层、服务层、数据访问层、工具类等目录
- GUI 页面、前后端交互、命令行入口、调度脚本
- API 调用、网络请求、任务调度、多线程、文件读写

## 5. 数据与数据库证据

- SQL 建表、插入、更新、查询脚本
- 数据库文件、迁移脚本、ORM 模型
- 实体、表、字段、索引、外键、约束、视图、存储过程

## 6. 界面与素材证据

- 截图图片、UI 原型、页面命名
- `images/`、`assets/`、`screenshots/` 等资源目录
- 若无图片，是否能从页面代码、菜单命名、功能流程推断占位图名称

## 7. 测试与验证证据

- `test`、`tests`、`spec` 目录
- 测试脚本、日志、断言、Mock 数据
- 手工测试记录、运行结果截图、异常处理说明

## 8. 章节证据优先级

题目驱动优先：

- 开发背景
- 研究意义
- 用户需求
- 应用场景
- 需求分析
- 项目目标

代码证据优先：

- 总体设计
- 详细设计
- 数据库设计
- 编码实现
- 运行界面
- 测试与问题分析

需求分析章节允许参考目录边界，但不能直接把代码功能列表改写为需求说明。

## 9. 需要特别避免的误判

- 不要把提示词模板里的示例功能写成项目已有功能
- 不要把通用课程要求写成项目已经完成的实现
- 不要把文件名猜测成真实模块职责
- 不要把数据库设计常识写成项目已经落地的表结构
- 不要把代码现状当成需求分析的唯一来源

## 10. 证据不足时的写法

如果某部分证据不足，可以：

- 优先从题目、课程要求和任务目标展开需求分析
- 在设计和实现部分保持抽象层次，不给出虚假的实现细节
- 对截图使用占位图方案
- 对测试写成“测试设计”或“验证方案”

不可以：

- 臆造接口、页面、字段、测试数据
- 写出不存在的运行结果
- 使用“根据代码分析可知”这类机械表达
